Per D-11 (incremental swap) and D-12 (no backfill), swap the Analytics page's primary data queries to read from the aggregates table. The swap targets (ordered by impact per RESEARCH.md recommendation):

**Swap 1: costByProvider -> api.aggregates.costByPeriod**

Replace the direct `useQuery(api.llm.costByProvider)` call with the aggregate equivalent:

```typescript
// OLD (remove):
const costByProvider = useQuery(api.llm.costByProvider) ?? {};

// NEW:
const costByProvider = useQuery(api.aggregates.costByPeriod, { period: "daily" }) ?? {};
```

The return shape is identical — `Record<string, number>` grouped by provider — so `totalCost` computation and `CostTrendChart` / `LlmAnalyticsPanel` props remain compatible.

**Swap 2: Add error trend aggregate query for ErrorRateTrend component**

Add a new query call and pass it to the ErrorRateTrend component:

```typescript
const errorTrend = useQuery(api.aggregates.errorTrendByPeriod, { period: "hourly" }) ?? [];
```

Pass `errorTrend` as a prop to `<ErrorRateTrend />` if it accepts external data. If ErrorRateTrend currently fetches its own data internally via `useQuery(api.analytics.errorRateTrend)`, leave it alone for now — it already has the archived filter from Task 2, and the component-level swap requires modifying ErrorRateTrend.tsx which is a separate concern. Document in the SUMMARY which components were swapped and which still use raw queries.

**Swap 3: Add event counts aggregate for the "Total Events" MetricCard**

const eventCounts = useQuery(api.aggregates.eventCountsByPeriod, { period: "daily" }) ?? {};
