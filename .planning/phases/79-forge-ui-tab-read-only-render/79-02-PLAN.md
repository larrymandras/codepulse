<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Convex cloud → React UI | forgeJobs prompt / capabilities / workspaceId / hostId strings (agent-emitted, untrusted) render into list cards and the metadata panel. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-79-03 | Tampering (XSS via job prompt / capabilities) | ForgeJobList card prompt, ForgeMetadataPanel capabilities block | mitigate | Render prompt and parsed-capabilities strings as JSX text children only — React default-escaping is relied upon. No `dangerouslySetInnerHTML`. `JSON.parse` of capabilities is wrapped in try/catch (parse failure falls back to displaying the raw string, never executes it). |
| T-79-04 | Information disclosure (PII in job prompt) | ForgeJobList / ForgeJobDetail prompt display | accept | Job prompts are operator-authored agent prompts shown to the same operator on an authenticated dashboard; the existing PrivacyContext masking convention is available but prompt content here is first-party. Low severity; no cross-tenant exposure (single-operator dashboard). |
| T-79-05 | DoS (oversized capabilities / prompt string) | metadata capabilities block | accept | Strings are bounded by what the forge daemon emits; long values truncate (workspaceId) or wrap (capabilities). No high-severity amplification. |
| T-79-SC | Tampering (supply chain) | npm installs | accept | No new packages — ScrollArea/Skeleton/Badge/lucide-react already present. No legitimacy gate required. |

No high-severity threats: read-only render, no mutations, no user-supplied input beyond list selection, React escaping relied upon for all untrusted strings.
</threat_model>
