**`sendAlertWebhook` internalAction** (per D-07, D-08):
- Args: `alertId: v.id("alerts"), attempt: v.number()`
- Implementation:
  1. `const alert = await ctx.runQuery(internal.alerts.getById, { id: args.alertId })`; if null return
  2. `const channels = await ctx.runQuery(internal.webhookDelivery.getNotificationChannels, {})`
  3. `const prefs = await ctx.runQuery(internal.webhookDelivery.getNotificationPreferences, {})`
  4. Check mute: `const muted = await ctx.runQuery(internal.alertMutes.isTargetMuted, { targetType: "alert", targetId: args.alertId })`; if muted, return
  5. Check delivery mode: `const mode = prefs[alert.severity] ?? "always"`; if `"dashboard_only"` or `"disabled"`, return; if `"digest"`, return (digest cron handles)
  6. Retry delays: `const RETRY_DELAYS = [5000, 30000, 120000]` (5s, 30s, 2m)
  7. Try: send to Discord (`fetch(channels.discordUrl, { method: "POST", headers: {"Content-Type": "application/json"}, body: JSON.stringify(buildDiscordPayload(alert)) })`); send to Slack similarly. On success: `await ctx.runMutation(internal.alerts.updateWebhookStatus, { id: args.alertId, status: "delivered", deliveredAt: Date.now() / 1000 })`
  8. Log each attempt to `webhookDeliveryLog` table via an internalMutation
  9. Catch: if `args.attempt < 3`: `await ctx.scheduler.runAfter(RETRY_DELAYS[args.attempt - 1], internal.webhookDelivery.sendAlertWebhook, { alertId: args.alertId, attempt: args.attempt + 1 })`; else: `await ctx.runMutation(internal.alerts.updateWebhookStatus, { id: args.alertId, status: "failed", attempts: args.attempt })`

**`logDeliveryAttempt` internalMutation**:
- Args: `alertId, channel, attempt, status, statusCode?, errorMessage?, sentAt`

**Upgrade `convex/alerts.ts` `evaluateInternal`** to support Phase 6 features:

1. **Query threshold overrides**: For each static rule being evaluated, query `agentConfigs` for key `"alert-rule-override:{rule.id}"`. If override exists, use override threshold and lookbackWindow instead of defaults.

2. **Evaluate custom rules**: After static rules, query `alertRuleCustom` for all enabled custom rules. For each custom rule, evaluate its conditions array against the relevant metric queries. Apply `conditionLogic` ("AND" = all conditions must be true, "OR" = any condition must be true). If `conditionGroups` exists, evaluate each group and combine with the top-level logic.

3. **Lookback window enforcement** (per D-03): When checking a condition, only count events within the lookback window. Map window strings to seconds: "5m"->300, "15m"->900, "30m"->1800, "1h"->3600, "24h"->86400. Filter events to `createdAt > (now - windowSeconds)`.
