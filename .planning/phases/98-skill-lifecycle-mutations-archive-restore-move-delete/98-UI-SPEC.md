# Phase 98 — UI Design Contract

> Visual and interaction contract for Phase 98 (Skill Lifecycle Mutations — archive / restore / move / delete). This is a bounded UI delta on the existing Skills surface, not a redesign: a new per-row overflow menu (⋯), two small dialogs (move-destination picker, permanent-delete type-to-confirm), and status-badge reuse for lifecycle command rows. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope boundary:** `SkillRow.tsx`, `ColdStorageView.tsx` gain a ⋯ menu. New: a lifecycle move-destination dialog (reuses `IntakeModal`'s workspace-picker pattern) and a permanent-delete `AlertDialog` (type-the-name confirm). Reuses `useIntake.ts`'s status-mapping/badge conventions for lifecycle command rows (no new hook family — see Phase UI Deltas). No drag-and-drop, no optimistic lane-move, no polished menu redesign — those are explicitly Phase 100 (D-08, deferred).

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps inside menu items and badges |
| sm | 8px | Menu item internal gap (`gap-2`), row-level status badge padding |
| md | 16px | Dialog/AlertDialog section padding (`p-4`), form field stacks (`gap-4`) |
| lg | 24px | Not newly introduced this phase |
| xl | 32px | Not newly introduced this phase |

Exceptions: the ⋯ trigger button reuses `SkillRow`'s existing icon-button shape (`p-1 rounded hover:bg-primary/20`, ~24px hit target visually) — but per the 8-point touch-target exception, the clickable area must be padded to at least 32px (add `min-w-8 min-h-8` or equivalent) since this is a genuinely new interactive control, not decoration. The existing hover-reveal icon buttons in `SkillRow` (Chat/Star/Pencil) are visual-only precedent, not a touch-target precedent — do not copy their tight padding for the new menu trigger.

## Color

Token-driven — reuse existing CSS vars, never hardcode hex. This phase introduces zero new colors; it composes the already-defined status palette onto new lifecycle states.

| Role | Token | Usage |
|------|-------|-------|
| Dominant (60%) | `--background` (zinc) | Page/row background, dropdown menu surface |
| Secondary (30%) | `--card` / `--border` (zinc) | Row dividers, menu separator, dialog surface |
| Accent (10%) | `--primary` (theme-driven, e.g. cyan) | Reserved for: the ⋯ trigger's hover/focus ring, "Restore" menu item's default (non-disabled) icon+label color, the move-destination dialog's confirm button fill |
| Success | `--status-ok` | Not used this phase (archive/restore success renders as a lane change via the rescan, not a success badge) |
| Warning | `--status-warn` | Shadow-blocked Restore tooltip icon, cold-collision inline error accent |
| Error / Destructive | `--status-error` | Permanent-delete `AlertDialog` destructive button + trash icon, "Delete" menu item label color, all lifecycle-command `failed` badges (reuses `RowStatusBadge`'s existing `failed` config verbatim) |
| Info | `--status-info` | Pending/executing lifecycle badge (reuses `RowStatusBadge`'s existing `executing` config verbatim) |

Accent reserved for: ⋯ trigger hover/focus state, non-destructive menu item default icon color (Restore/Move), move-dialog confirm CTA fill. Body text, menu item labels for Archive/Move, and row descriptions stay neutral (`text-foreground`/`text-muted-foreground`).

## Copywriting Contract

House honesty rule applies: no false success, no bare generic labels, every destructive action names what it destroys.

| Element | Copy |
|---------|------|
| Primary CTA (⋯ trigger) | No visible label — icon-only `MoreVertical` (Lucide), `aria-label="Skill actions for {displayName}"` |
| Menu item: Archive | **"Archive"** (icon: `Archive`) — no confirmation dialog per Claude's Discretion (reversible; D-07 "no throwaway"); a light one-line toast/inline note is acceptable, not required |
| Menu item: Restore | **"Restore"** (icon: `ArchiveRestore` or `Undo2`) — disabled + tooltip when shadow-blocked (see below) |
| Menu item: Move (active rows only) | **"Move to Project…"** / **"Move to Global…"** (icon: `FolderInput`) — label names the *destination*, not the generic verb, since the row's current scope determines which single direction is offered |
| Menu item: Delete (cold-storage rows only) | **"Delete Permanently"** (icon: `Trash2`, `--status-error` colored) — never bare "Delete"; must always say "Permanently" since this is the one truly destructive action in this phase (D-05) |
| Shadow-blocked Restore tooltip (D-09) | **"Shadowed by an active {scope} skill named "{name}" — archive it first."** |
| Cold-collision error (D-02) | **"A dormant copy of "{name}" already exists in Cold Storage. Rename or delete it first, then archive again."** |
| Move-target collision error (D-02) | **"A skill named "{name}" is already active in {scope}. Rename or remove it first, then move again."** |
| Shadow-block hard-refuse error (D-03, if a restore somehow reaches the daemon despite the client pre-check) | **""{name}" is active in {scope} — archive it first."** |
| Move-destination dialog title | **"Move "{name}" to Project"** (title names the skill and direction, never generic "Move skill") |
| Move-destination dialog body | **"Choose a synced workspace. The skill moves immediately once you confirm."** |
| Move-destination confirm button | **"Move skill"** |
| Move-destination cancel button | **"Cancel Move"** (bare "Cancel" banned — this is a new confirm surface) |
| Permanent-delete AlertDialog title | **"Delete "{name}" permanently?"** |
| Permanent-delete AlertDialog body | **"This removes the file from disk. There is no undo and no git history for .claude/ — type the skill name to confirm."** |
| Permanent-delete confirm input placeholder | **"Type "{name}" to confirm"** |
| Permanent-delete confirm button | **"Delete Permanently"** (stays disabled until typed name matches exactly; `--status-error`/destructive `AlertDialogAction` variant) |
| Permanent-delete cancel button | **"Cancel"** is acceptable here ONLY because `AlertDialogCancel`'s default shadcn label is already the established house pattern for this exact primitive elsewhere in the app (e.g. existing stop/discard dialogs) — if plan-phase finds no such precedent, use **"Keep skill"** instead |
| Offline-daemon lifecycle row (LIFE-06) | Reuses `IntakeStatusBadge`'s existing "queued, will expire" copy pattern verbatim — e.g. **"Queued — expires in {mm:ss} unless a daemon claims it"** (mirrors `IntakeModal`'s existing offline-host inline warning copy) |
| Lifecycle command failed (generic) | **"{Action} failed: {daemon's real error message}. Nothing changed on disk."** — the "Nothing changed" clause is mandatory per the atomic-mutation guarantee (D-01/D-02), mirrors Phase 97's "Nothing was written" pattern |
| Destructive confirmation summary | Archive: no dialog (reversible). Delete: `AlertDialog` + type-the-name gate (D-06). Restore/Move: no confirmation dialog — shadow-block and collision are pre-flight validation errors, not confirmation gates |

## Phase UI Deltas — Prescriptive Specs

### 1. Per-row ⋯ menu (D-07)

Add a `DropdownMenu` trigger to `SkillRow.tsx` (visible in both the normal grid and `ColdStorageView`), positioned in the existing hover-reveal action cluster (alongside Chat/Star/Pencil icons) but **always visible** (not hover-only) — lifecycle actions are not decoration, they must be discoverable without a hover-capable pointer. Menu contents are scope-gated:

- **Active (global/project) row:** Archive, Move to {other scope}…
- **Dormant/cold-storage row:** Restore, Delete Permanently
- Never render an action the current scope doesn't support (e.g. no "Archive" on an already-dormant row, no "Delete Permanently" on an active row per D-05).

This is a simple, unstyled-beyond-tokens `DropdownMenu` — explicitly NOT the polished/drag-integrated menu Phase 100 builds. Do not add animation flourishes, submenus, or icons beyond the single Lucide icon per item specified in the Copywriting Contract.

### 2. Lifecycle command row states (D-08)

Reuse `useIntake.ts`'s `IntakeRowStatus` model (`pending | queued | executing | done | failed | expired`) and `IntakeStatusBadge.tsx`'s existing badge configs **verbatim** — do not invent a parallel status enum or a new badge component for lifecycle commands. The row itself does not move lanes until the daemon's rescan (`syncInventory`) lands; until then it shows its current lane with a small status badge overlay (reuses `RowStatusBadge`'s `executing`/`queued`/`failed` visual configs). No optimistic lane-move animation this phase (Phase 100 / UX-03).

### 3. Shadow-blocked Restore (D-09)

When `isShadowing(skill)` (from `src/lib/skills.ts:26`) is true for the target scope, the "Restore" menu item renders `disabled` with a `Tooltip` (not a native `title` attribute — use the existing `tooltip.tsx` primitive for consistent styling) showing the exact copy from the Copywriting Contract. The daemon's server-side re-check (D-04) is a backstop only — the UI must never let a click reach the daemon in the client-detectable shadow case.

### 4. Move-destination dialog

A small `Dialog` (not a full modal port of `IntakeModal`) triggered by "Move to Project…" — reuses `IntakeModal`'s workspace `Select` pattern exactly (synced-workspace list, "No workspaces synced from this host yet" empty state copy carried over verbatim). "Move to Global…" (project → global direction) needs no destination picker — global has no sub-target — so it skips the dialog entirely and enqueues immediately with a lightweight optimistic status badge (no confirmation, mirrors Archive's no-dialog treatment since Global is a single fixed destination).

### 5. Permanent-delete confirm (D-06)

`AlertDialog` (not `Dialog` — this is shadcn's destructive-action primitive) with an `Input` for the type-to-confirm gate. The `AlertDialogAction` (destructive variant, `--status-error`) stays `disabled` until the input's trimmed value exactly (case-sensitive) matches the skill's real name. No autofocus-and-select-all shortcut that could be bypassed by a stray paste — require deliberate typing or an explicit paste-and-verify (either is fine; do not pre-fill the input).
