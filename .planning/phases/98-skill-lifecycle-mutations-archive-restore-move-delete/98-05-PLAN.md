<objective>
Close the single diagnosed Phase 98 gap: moving (or deleting) the LAST skill out of a project workspace leaves a stale `claude-code:project:<key>` row in Convex forever, which then renders as a multi-scope skill and wrongly disables Archive/Move in the lifecycle menu.

Root cause (already verified live — do NOT re-diagnose): `computeSkillPrunes` (convex/skillSync.ts:44) skips pruning for any origin absent from the incoming snapshot. The snapshot only carries skills that still exist, so an origin whose workspace became EMPTY is indistinguishable from an origin that was never scanned — its stale rows are never deleted.

Fix (validated during diagnosis, cross-repo forge + codepulse):
- **Producer (forge):** `buildSkillSnapshot` declares the origins it COVERED — a `scannedOrigins` manifest built from the two home roots plus every REACHABLE walked workspace, so a covered-but-empty origin is now distinguishable from an unscanned one.
- **Consumer (codepulse):** `computeSkillPrunes` gains an optional `scannedOrigins` parameter; when present it prunes rows for any DECLARED origin even when that origin has zero incoming skills. `registry.syncInventory`/`syncFullInventory` pass `snap.scannedOrigins` through and relax the length gate.
- **Backward compatibility:** a snapshot WITHOUT the manifest keeps today's conservative behavior (absent origins untouched). Reachability gating means a transiently-unmounted workspace (e.g. the G: Google Drive mount) is never declared and thus never pruned.

Purpose: Make the registry rescan reconcile emptied origins to host truth without ever over-pruning on a transient unmount.
Output: Producer manifest + consumer prune change + wiring, unit-tested on both sides including the regression pair.
</objective>
