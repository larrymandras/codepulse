## Typography

Source: Phase 71 UI-SPEC §2.7 type scale. All values are live de-facto tokens.

**Exactly two weights:** 400 (regular) and 600 (semibold). No medium (500) — collapsed to keep the contract at two weights.

| Role | Size | Weight | Line Height | Font | Usage |
|------|------|--------|-------------|------|-------|
| Body | 14px (`text-sm`) | 400 (regular) | 1.5 | Geist | Form labels, workspace names, prompt text, error copy |
| Subheading | 12px (`text-xs`) | 400 (regular) | 1.4 | Geist | Field labels inside modal, metadata keys |
| Label | 12px (`text-xs`) | 600 (semibold) | 1.4 | JetBrains Mono + `uppercase tracking-widest` | Status badge text, section chrome, helper copy |
| Heading | 16px (`text-base`) | 600 (semibold) | 1.2 | Geist | Modal title ("Launch Job"), confirm dialog title ("Stop this job?") |

Rules:

- Modal title: `text-base font-semibold` (DialogTitle default) — Heading role
- Form field labels: `text-sm` (Body role, weight 400) — field labels do NOT use a medium weight; the NewJobModal source's `font-medium` is intentionally dropped to hold the two-weight contract
- Helper / disabled copy: `text-xs text-muted-foreground` (Subheading role)
- Error inline copy: `text-sm text-destructive` (Body role)
- Stop confirm body: `text-sm` at 1.5 line height — readability for destructive warning

## Visual Hierarchy / Focal Point

**Primary visual anchor:** the "Launch Job" button (emerald fill, `variant="default"`) in the
ForgeJobList header draws the eye first on the `/forge` screen — it is the only emerald-filled
element in the otherwise zinc/neutral list surface. Within the launch modal, the "Launch Job"
submit button in the DialogFooter is the terminal focal point (the single emerald action,
bottom-right, where the eye lands after filling the form). On ForgeJobDetail, the
destructive-red "Stop" button is the focal point for a running job — the only saturated control
in the header.

Hierarchy order on the primary screen:
1. "Launch Job" toolbar button (emerald) — primary action
2. Selected job row (emerald focus ring / accent) — current context
3. Status badges (semantic color) — at-a-glance state
4. Job metadata / prompt text (muted) — supporting detail

## New Surfaces — Interaction Contract

### Surface 1: ForgeLaunchModal

**Trigger:** "Launch Job" toolbar button in `ForgeJobList` header. Renders as a shadcn `Dialog`.

**Width:** `max-w-[520px]` — match NewJobModal source exactly.

**Layout (top to bottom, `gap-4` between sections):**

```
DialogHeader
  DialogTitle: "Launch Job"

[Host Picker]           ← NEW (D-08)
  Label: "Host"
  Select: online host pre-selected; stale/offline hosts disabled with "(offline)" suffix
  Online indicator: 8px emerald dot beside online host name
  Loading state: Skeleton w-[200px] h-9 while hosts load

[Agent Picker]          ← ported, unchanged
  Label: "Agent"
  Select: Codex (default) | Claude Code | Antigravity (disabled)
  Helper: "Antigravity is disabled — PTY spike failed on this machine" text-xs muted

[Workspace Picker]      ← ported, inline-create REMOVED (D-07)
  Label: "Workspace"
  Select: lists forgeWorkspaces filtered to selected hostId
  Empty state inline: "No workspaces synced from this host yet." text-xs muted
  Note: no "+ New workspace" button — D-07 drops inline creation

[Mode Segmented Control] ← ported, unchanged
  Two segments: Goal (default) | Chat
  Active: bg-primary text-primary-foreground

[Prompt Textarea]       ← ported, unchanged
  Label: "Prompt" (not required for chat mode)
  Placeholder: "Describe what you want the agent to do…"
  min-h-[96px] max-h-[192px] resize-y

[Advanced Collapsible]  ← ported, dangerous-mode section REMOVED (D-06)
  Trigger: "▸ Advanced" / "▾ Advanced" text-sm muted
  Contents:
    Model (Claude agent): Select dropdown — CLAUDE_MODELS list, default claude-opus-4-8
    Model (Codex agent): Input text, placeholder "gpt-5.5"
    Max turns: Input number, min 1 max 500, default 50

[Submit Error]          ← shows only on failure
  text-sm text-destructive

DialogFooter
  Button "Cancel" variant="outline"
  Button "Launch Job" / "Send" variant="default" (emerald) — disabled until: host selected + workspace selected (if workspaces exist) + (agent=claude → model set)
  Submitting state: button disabled, label "Launching…" with Loader2 spinner inline
```

**Submit enabled condition:**

- Host selected (and online)
- Workspace selected (if `listWorkspaces` for that host returned results)
- If agent = `claude`: model is non-empty
- Not currently submitting

**On submit:** calls `useMutation(api.forge.enqueueLaunch).withOptimisticUpdate(...)`. Modal closes immediately. Optimistic "Queued" row appears in ForgeJobList.

### Surface 2: ForgeStopConfirmDialog

**Trigger:** "Stop" button on `ForgeJobDetail` header. Button renders only when `job.status === "running"`.

**Pattern:** shadcn `AlertDialog` (already in `src/components/ui/alert-dialog.tsx`) — D-03 mandates a confirm gate, not a one-click action.

**AlertDialogContent layout:**

```
AlertDialogHeader
  AlertDialogTitle: "Stop this job?"
  AlertDialogDescription:
    "This will immediately kill the agent process (taskkill /T /F).
     Any work in progress — not yet promoted to the workspace — will
     be discarded. This cannot be undone."

AlertDialogFooter
  AlertDialogCancel: "Cancel"
  AlertDialogAction:  "Yes, stop the job"   ← variant="destructive" by convention
```

**Stop button on ForgeJobDetail header:**

```
Button variant="destructive" size="sm"
  When job.status === "running":       "Stop"           (active, triggers AlertDialog)
  When stopping_pending (D-04):        "Stopping…"      Loader2 inline, disabled
  When job.status === "stopped":       hidden            (terminal state)
```

**Stopping pending state (D-04):**

After confirm, the Stop button flips to `"Stopping…" disabled` immediately (local optimistic state on the button only — the `forgeJobs` status badge does NOT change optimistically). The badge transitions to `stopping_pending` only after the reactive query delivers `status:"stopping_pending"` from the server, or stays `running` until `stopped` arrives. The button's local disabled state acts as the UX feedback. See anti-pattern note in RESEARCH.md §Anti-Patterns.

### Surface 3: ForgeJobList — Pending Rows and Toolbar

**Launch toolbar button:**

```
ForgeJobList header (right side, beside filter/sort controls if any):
  Button variant="default" size="sm" icon=Rocket className="gap-2"
    "Launch Job"
  Gated: rendered only when Clerk identity is present (useClerk().user !== null)
  When Clerk not configured: button absent (fail-closed; no anonymous launch)
```

**Optimistic "Queued" pending row (D-10):**

Pending rows are command-sourced (`forgeCommands` where status is `queued|executing`) and
render as a distinct row type in ForgeJobList. They are **not** real `forgeJobs` rows.

```
Row visual:
  Left border: border-l-2 border-primary (emerald left bar, distinguishes pending from real)
  AgentIcon: same as real rows
  Status badge: ForgeStatusBadge status="pending" (new variant, see color table)
  Prompt: prompt text or "—" if blank
  Host: ForgeHostBadge for selected hostId
  Time: "Just now" (no server timestamp yet)
  Detail pane: clicking a pending row shows a minimal detail:
    "Queued — waiting for Forge daemon on {hostId}…"
    No metadata panel (no real forgeJobs row yet)

Failure flip (D-11):
  If forgeCommands.status → "failed" or "expired":
    Row keeps the border-l but switches to border-destructive
    Status badge: ForgeStatusBadge status="failed" or "expired"
    Error text appears below prompt: text-xs text-destructive {error message}
    Detail pane shows: "Command failed: {error}" or "Command expired — daemon was offline."
```

**Reconciliation:** Once `forgeCommands.resolvedForgeJobId` is set, the pending row is
filtered out and the real `forgeJobs` row takes over. No duplicate rendering.

### Surface 4: ForgeStatusBadge — New Variants

Two new status string values added (must be wired into the `STATUS_MAP` in `ForgeStatusBadge.tsx`):

| New status | Label | className | Icon | Animated |
|------------|-------|-----------|------|----------|
| `pending` | "Queued…" | `bg-zinc-800/60 text-primary` | `Loader2` | `animate-spin` |
| `stopping_pending` | "Stopping…" | `bg-amber-900/40 text-[var(--status-warn)]` | `Loader2` | `animate-spin` |
| `expired` | "Expired" | `bg-zinc-800/30 text-zinc-600` | `Clock` | none |

The existing `queued` variant (zinc-400, `Clock`, no animation) maps to the Forge-native
`queued` status. The new `pending` variant (emerald text, spinning) maps to the cloud-initiated
optimistic state — visually distinct from Forge-native queued.
