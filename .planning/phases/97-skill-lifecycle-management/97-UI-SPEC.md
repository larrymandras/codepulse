# Phase 97 — UI Design Contract

> Visual and interaction contract for the bounded frontend delta in Phase 97 (Real Skill Intake & Daemon Foundation). This is primarily a daemon/backend phase — this contract covers ONLY the 4 UI deltas on the already-built intake surface, not a re-spec of the whole `IntakeModal`/`IntakeSheet` flow. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope boundary:** `IntakeModal.tsx`, `IntakeReportView.tsx`, `IntakeSheet.tsx`, `IntakeStrip.tsx`, `IntakeStatusBadge.tsx`, `useIntake.ts`, `useIntakeFeed.ts` already exist and ship today as a dry-run surface. This contract specifies the deltas required to make the SAME surface tell the truth about a REAL install (INTAKE-01..04). No new page, no new nav entry, no new top-level component family.

## Copywriting Contract

The core copy change this phase: the surface currently frames every action as a **dry run** ("Validate skill", "Validation only — nothing is written"). Once INTAKE-01..04 land, submitting actually writes to the host filesystem — continuing to say "Validate" is a false-success framing violation (house honesty rule). All dry-run language must be replaced with install language.

| Element | Copy |
|---------|------|
| Primary CTA (single) | **"Install skill"** — replaces `IntakeModal.tsx` `DialogTitle` ("Validate skill") and the submit `Button` label |
| Primary CTA (batch, N selected) | **"Install {N} skills"** — replaces "Validate {N} skills" (both submit-button and loading-state variants at ~L500-511) |
| Modal footer note (replaces "Validation only — nothing is written", ~L486) | **"Installs to the selected host and destination."** — a real-write disclosure, not a dry-run disclaimer. If the selected destination is empty, this line may be omitted or shown greyed until a destination is chosen (Claude's discretion — either is acceptable, pick one and apply consistently) |
| Empty state heading (`IntakeSheet`, unchanged structurally) | "No intake commands yet" (no change — already honest) |
| Empty state body (`IntakeSheet`, ~L63) | **"Drop a SKILL.md or paste a GitHub URL with "Install skill" — reports appear here."** — update the quoted button-name reference from "Validate skill" to match the renamed CTA |
| Fan-out confirm step (D-06) — see Phase UI Deltas §2 | Heading: **"{N} skills found at this path"**; body: **"Review the list below, then confirm to install all selected skills to {destination}."**; Confirm button: **"Install {N} skills"** (same label as batch CTA above — one mental model, not two); Cancel: **"Cancel"** |
| Failed install / collision error (D-07) — see Phase UI Deltas §3 | Must be actionable, never a bare exit code or stack trace. Canonical pattern: **"A skill named "{name}" already exists at {destination}. Choose a different destination, or remove the existing skill first."** Generic (non-collision) failure pattern: **"Install failed: {daemon's real error message}. Nothing was written."** — the "Nothing was written" clause is mandatory per INTAKE-04's no-partial-state guarantee whenever the daemon confirms a clean rollback |
| Destructive confirmation | **None in this phase.** Archive/restore/move/delete (LIFE-01..06) are out of scope — Phase 98. The fan-out confirm above is a "confirm-before-write" gate, not a destructive-action confirmation, and does not need an `AlertDialog`'s destructive-red treatment — use the standard `Dialog`/`AlertDialog` with a `variant="default"` confirm button |

**Global find-and-replace obligation:** grep the intake surface for the literal strings `"Validate skill"`, `"Validate {N} skills"` / `` `Validate ${...} skills` ``, and `"Validation only"` before considering this phase's UI deltas done — every occurrence must be updated to the install-language equivalents above. (Known sites: `IntakeModal.tsx` DialogTitle + submit button ×2 variants + footer note; `IntakeSheet.tsx` empty-state body.)

## Phase UI Deltas — Prescriptive Specs

### 1. Real execution/validation report (INTAKE-04)

`IntakeReportView.tsx` already renders `VerdictBadge` + `SeverityBadge` + a findings `Table` (rule/severity/file:line/reason) from `row.report`. **No structural change required** — the component already assumes a real `ReportEnvelope` shape (`verdict`, `summary`, `findings[]`). The only change this delta needs:

- Confirm the daemon's real report conforms to the existing `ReportEnvelope` contract the component already reads (`verdict`, `summary: {error, warning, info}`, `findings: [{rule_id, severity, path, line, message}]`). If plan-phase research finds the real `skill-intake` report shape diverges, an adapter belongs in `convex/forge.ts` (server-side), not a new UI branch — keep `IntakeReportView` shape-agnostic of dry-run vs real.
- The `capAckReport`-truncation branch ("Report too large to store — run the CLI command below") is unchanged and still correct for an oversized real report.
- The CLI-command reconstruction already includes `--write` (it was written assuming a future real invocation) — no copy change needed there.

### 2. Fan-out confirm-first UX (D-06)

**Reuse `SkillCollectionPicker.tsx` as the canonical candidate-set surface** — it already renders exactly what D-06 asks for: "N skills found" heading, a `Checkbox` "Select all" + per-skill checkbox list (name + path), and the modal's own `DialogFooter` Cancel/Submit pair already serves as the confirm/cancel affordance. Do not build a second list component.

Two possible technical shapes (plan-phase must confirm which applies — spec covers both):

- **If resolution stays client-side** (today's `useGithubTreeScan` GitHub-tree scan, unchanged): the existing Scan → checkbox-list → submit flow already satisfies "resolve → confirm the set → install." Only the button/title copy changes (see Copywriting Contract).
- **If `skill-intake`/daemon requires a server-side resolve round-trip** separate from write (the open question flagged in 97-CONTEXT D-06): insert a **"Resolving…" loading state** between submit-click and candidate-list render — reuse the existing `Loader2` spin pattern already used on the submit button, with the label "Resolving skills at this path…". Once resolved, render the SAME `SkillCollectionPicker`-style checkbox list (do not invent a second visual pattern) inside the existing `Dialog`, then require an explicit second confirm click before any `enqueueIntake` call fires. Cancel returns to the un-resolved form state without submitting anything.

In both shapes: the confirm action must NEVER auto-submit on resolve — an explicit user click (Install button) is required, per D-06's "prevents a surprise bulk-install" rationale.

### 3. Collision / failed-install error state (D-07)

A same-name collision at the destination scope must render as a **clean failed install with an actionable error — never a silent overwrite.**

- **Preferred shape** (if the daemon surfaces collision as a validation-time rejection): row lands with `status: "done"`, `report.verdict: "reject"`, and a `findings[]` entry (e.g. `rule_id: "collision"` or similar, `severity: "error"`). This renders through the **already-existing** `IntakeReportView` findings table — `VerdictBadge` shows red "Reject", the table row shows the collision finding. No new component. The finding's `message` field must use the exact actionable copy pattern from the Copywriting Contract above (name + destination + next step), not a raw daemon exception string.
- **Fallback shape** (if the daemon instead terminates the command as `status: "failed"` with a plain `error` string — e.g. an execution-layer, not validation-layer, failure): `IntakeSheet.tsx`'s existing `row.status === "failed"` branch (`"Failed: {row.error}"`, ~L146-148) must receive the same actionable string — the CONTRACT is on the copy, not the status field. Whichever branch the daemon's real behavior lands in, the rendered text must satisfy the actionable-error pattern above.
- Either way: **no new visual treatment is needed** — `RowStatusBadge`'s existing `failed` config (`--status-error` red, `XCircle` icon) and `VerdictBadge`'s existing `reject` config already carry the correct color/iconography. This delta is a **copy and data-shape contract**, not a new UI surface.

### 4. Optimistic intake row lifecycle states (real outcomes)

`useIntake.ts`'s `mapIntakeStatus` (`pending | queued | executing | done | failed | expired`) and `IntakeStatusBadge.tsx`'s existing badge maps already cover every state this phase needs:

- `pending` (client-only, pre-round-trip) → unchanged
- `queued` → unchanged, countdown label unchanged
- `executing` → unchanged (`--status-info`, already anticipates a live daemon)
- `done` → now carries a REAL verdict (`admit` = real success, `reject`/`error` = real failure) via the already-wired `VerdictBadge`
- `failed` → now carries a REAL transport/execution error (was previously unreachable in the dry-run world except client-side enqueue failures)
- `expired` → unchanged, still the "no daemon claimed this command" case

**No new lifecycle state is required.** Do not add a 7th status. If plan-phase discovers the daemon needs a distinct "installing" vs "validating" sub-state within `executing`, collapse it back to the existing `executing` badge rather than adding a new chip — the existing 6-state model already satisfies INTAKE-04's success/failure/no-partial-state requirements.
