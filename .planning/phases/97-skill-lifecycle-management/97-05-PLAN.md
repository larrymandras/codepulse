<objective>
Turn the daemon's structured write-refusal signal into the actionable, house-copy error the UI-SPEC mandates — server-side — and put that copy where the UI actually reads it on a failed row: `row.error`.

Purpose: Plan 01's `mapExitCodeToResult` emits a machine-readable `error` string (`write-refused:<kind>:<raw>` for exit 4-7, `post-placement-warning:<kind>:<raw>` for exit 8-9) while keeping the CLI `report` verbatim (forge invariant D-P8-10 forbids the daemon reshaping the report). This plan adds the Convex-side adapter in `ackCommand` that maps that signal to (a) a synthesized `findings[]` entry + verdict flip AND (b) a matching actionable `error` string — because `IntakeSheet.tsx`'s failed branch renders ONLY `row.error` (it gates `IntakeReportView`/`report.findings` behind `status === 'done'`), so composing into findings alone would leave the raw internal token on screen (BLOCKER-1).

Output: a pure `synthesizeWriteRefusalReport` helper returning `{ report, error }` + its wiring into `ackCommand` before `capAckReport`, with unit coverage per refusal kind.

**Decision (locked here, Open Question 2):** adapter is Convex-side, not daemon-side. Rationale: `forge/src/emit/intake-types.ts` freezes D-P8-10 ("the daemon never synthesizes a fake report"); synthesis must therefore happen after the wire, and UI-SPEC states the adapter "belongs in convex/forge.ts (server-side), not a new UI branch."
</objective>
