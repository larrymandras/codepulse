## Summary

Phase 7 is a backend-heavy intelligence layer built almost entirely on infrastructure established in Phases 5 and 6. The aggregates pipeline (daily cost/error/event rows), alert creation mutations, webhook delivery, and the `agentConfigs` key-value config store are all in place and ready to extend. The primary new capabilities are: (1) a moving-average forecast computed from daily aggregates, (2) LLM-generated briefings triggered by session completion and daily cron, and (3) statistical anomaly detection that feeds the existing Phase 6 alert pipeline.

The LLM integration pattern is well-established in `convex/insightsChat.ts` — an `internalAction` that calls an OpenAI-compatible API with env vars (`OPENAI_API_KEY`, `OPENAI_BASE_URL`, `OPENAI_MODEL`). The new requirement is a dual-provider (primary + backup) config stored in `agentConfigs`, with the LLM caller reading those keys at runtime. Crons cannot call actions directly; the established pattern is cron → `internalMutation` → `ctx.scheduler.runAfter(0, internal.module.internalAction)` — exactly as `alerts.ts` does for webhook delivery.

The Briefings page currently contains entirely static template data (AGENT_MAP, SCHEDULE, BRIEFING_SECTIONS, BRIEFING_TEMPLATES constants — zero Convex queries). The entire page needs to be replaced with a dynamic feed backed by a new `briefings` table. Memory.tsx has a clean StatCard + tab system that can be extended with a fourth "Quality" tab with minimal surgery.

**Primary recommendation:** Build in five functional groups — (1) schema additions, (2) forecast computation query, (3) LLM briefing + digest pipeline, (4) anomaly detection cron + alert creation, (5) memory quality evaluation cron + UI. Each group is independently deployable with no cross-group dependencies within the phase.

### Pattern 3: Moving Average Forecast from Daily Aggregates

The `aggregates` table has rows with `metric_type="cost"`, `period="daily"`, `bucket_start` (epoch seconds UTC midnight), `value` (cost for that day), `dimensions: { provider, model }`.

```typescript
// Source: convex/aggregates.ts — costByPeriod query pattern
// Fetch last N daily cost rows, sum across providers per day:
const rows = await ctx.db
  .query("aggregates")
  .withIndex("by_type_period_bucket", (q) =>
    q.eq("metric_type", "cost").eq("period", "daily").gte("bucket_start", cutoff)
  )
  .collect();

// Group by bucket_start to get daily totals
const dailyTotals = groupByBucket(rows); // sum values per day

// Simple moving average:
const windowDays = 7; // or 14 — use 7 for sparse data (<30 days), 14 for dense
const recent = dailyTotals.slice(-windowDays);
const avgDaily = recent.reduce((s, v) => s + v, 0) / recent.length;

// Projections:
const projectedDaily = avgDaily;
const projectedWeekly = avgDaily * 7;
const projectedMonthly = avgDaily * 30;
```

**Window size recommendation:** Use 7-day window when fewer than 30 days of data exist; switch to 14-day when 30+ days available. The longer window smooths outliers better but requires more history. [ASSUMED — based on standard signal processing practice; no project-specific data density verified]

### Pattern 4: Z-Score Anomaly Detection

```typescript
// Source: decision D-09 (CONTEXT.md) + standard statistics
function computeZScore(value: number, values: number[]): number {
  const mean = values.reduce((s, v) => s + v, 0) / values.length;
  const variance = values.reduce((s, v) => s + Math.pow(v - mean, 2), 0) / values.length;
  const stdDev = Math.sqrt(variance);
  return stdDev === 0 ? 0 : (value - mean) / stdDev;
}

// Rolling window: last 14 or 30 daily values for baseline
// Flag: |zScore| >= 2 → warning; |zScore| >= 3 → critical
// Alert creation uses existing api.alerts.create mutation
```

**Rolling window recommendation:** 14-day window balances recency and stability. For latency, use hourly aggregates (last 24-48 hours). [ASSUMED — standard practice]

async function callLLMWithFallback(
  ctx: ActionCtx,
  prompt: string

  const primaryConfig = await ctx.runQuery(internal.briefings.getLLMConfig, { key: "intelligence.llm_primary" });

  } catch (err) {
    if (!backupConfig) throw err;
    return await callProvider(backupConfig, prompt);
  }
}

### Anti-Patterns to Avoid

- **Calling `action` directly from a cron:** Convex crons can only schedule `internalMutation` or `internalAction` — not public `action`. Always use `internal.*` references for cron-scheduled work.
- **Reading agentConfigs inside an internalMutation that calls scheduler:** `internalMutation` cannot call `ctx.runQuery` for `internalQuery`; it can only use `ctx.db` directly. Read config values directly from `ctx.db` in mutations.
- **Computing forecasts on every dashboard load:** Forecast computation should be a cached/stored value (either a Convex query with memoization or a periodic cron-stored result), not recomputed on each client request. Given daily aggregates are pre-computed, a `query` that reads them is fine — Convex caches query results reactively.
- **Using `take(N)` for large sliding windows:** The aggregates table uses `withIndex` + `gte` for time-ranged queries. Always use index-based range queries, not `take` + filter, to avoid full table scans.
- **Storing API keys as plain strings in briefing content:** API keys stored in `agentConfigs` must never appear in briefing narratives or log output.

## Common Pitfalls

### Pitfall 1: Session Briefing Triggered Multiple Times
**What goes wrong:** If session end fires a mutation that schedules a briefing action, and the session status flips multiple times (e.g., "completed" set then re-patched), multiple briefings get generated for the same session.
**Why it happens:** Convex mutations are transactional but the trigger (session status change) may happen in multiple codepaths.
**How to avoid:** Before scheduling the briefing action, check `briefings` table for an existing entry with the `sessionId`. If found, skip. Use an idempotency guard in the internalMutation.
**Warning signs:** Duplicate entries in the briefings feed for the same session.

### Pitfall 2: Daily Aggregate Gaps Skew Moving Average
**What goes wrong:** If Ástríðr was idle for 3+ days, those days have zero cost rows in `aggregates`. Including them as zero in the moving average dramatically underestimates the forecast.
**Why it happens:** `rollupDaily` only creates rows for days that had hourly aggregates. Zero-activity days have no rows.
**How to avoid:** When computing the moving average, query the last N days by calendar date (generate the expected bucket_start values for each day), and fill missing days with zero OR exclude them from the denominator. Decision: fill with zero for budget conservatism.
**Warning signs:** Forecasts dropping to near-zero after a weekend of inactivity.

### Pitfall 3: LLM Context Window Overflow on Session Briefings
**What goes wrong:** A long session (hundreds of events) generates more context than the model's context window allows.
**Why it happens:** Passing raw event payloads to the LLM without truncation.
**How to avoid:** Pass summarized event data — counts by type, top tool names, total cost, error count, session duration — not raw event payloads. The briefing prompt should work with structured summaries, not raw logs.
**Warning signs:** LLM API 400 errors with "context length exceeded" messages.

### Pitfall 4: Anthropic API Compatibility
**What goes wrong:** Anthropic's REST API does not perfectly mirror OpenAI's `/v1/chat/completions` endpoint for all parameters.
**Why it happens:** Anthropic uses `anthropic-version` header requirement and has different message format constraints.
**How to avoid:** For the backup provider, use a dedicated code path that constructs an Anthropic-native request if `provider === "anthropic"`, rather than assuming full OpenAI compatibility at the HTTP level.
**Warning signs:** 400/422 errors from Anthropic with "missing anthropic-version header" or "invalid request body".

### Pitfall 5: Memory Quality Contradiction LLM Costs
**What goes wrong:** Running LLM-based contradiction detection on all memory pairs is O(n²) and can generate significant LLM costs.
**Why it happens:** `episodicEvents` may have hundreds of entries; comparing all pairs is expensive.
**How to avoid:** Batch only recently added or recently updated memories (e.g., last 24h), not the full corpus. Limit to comparing memories of the same `agentId`. Cap batch size at ~20 pairs per cron run.
**Warning signs:** LLM cost spikes visible in the Analytics page immediately after the memory quality cron fires.

### Pitfall 6: Briefings Page Complete Rewrite Risk
**What goes wrong:** The current Briefings.tsx has no Convex queries — it's 100% static data. A partial rewrite risks leaving dead code (AGENT_MAP, SCHEDULE, BRIEFING_SECTIONS, BRIEFING_TEMPLATES) that confuses future readers.
**Why it happens:** Static template data is interleaved with JSX render logic.
**How to avoid:** Delete all static data constants at the top of the file; replace with `useQuery` calls. The new page should have zero hardcoded briefing content.
**Warning signs:** TypeScript errors from unused types after static data removal.

## Open Questions (RESOLVED)

1. **Where does session status flip to "completed"?** -- RESOLVED
   - **Answer:** `convex/sessions.ts` exports `markCompleted` mutation (line 40). Called from `convex/ingest.ts` line 103 when `eventType === "session_end" || eventType === "session_stop"`. Also called from `ingest.ts` line 174 for Claude Code hook session-end events.    - **Plan 03 action:** Add `ctx.scheduler.runAfter(0, internal.briefings.onSessionCompleted, { sessionId })` in `convex/ingest.ts` after the `markCompleted` call at line 103 (and line 174), when status === "completed". This avoids modifying the sessions module.

2. **Does `episodicEvents` track deduplication flags?** -- RESOLVED
   - **Answer:** No. `episodicEvents` has only `eventType: "memory_stored" | "memory_recalled" | "memory_pruned"` with no `isDuplicate` boolean field. The schema (line 420) confirms no dedup-specific flag.
   - **Dedup rate computation:** Use ratio of `memory_pruned` events to `memory_stored` events as the deduplication rate proxy. This is the approach Plan 05 already uses (`computeDeduplicationRate(totalStored, prunedCount)`).

3. **Activity changelog event source** -- RESOLVED
   - **Answer:** Rule-based template grouping (no LLM cost). The `events` table has `eventType`, `toolName`, `filePath` per session. Plan 03 implements `groupActivityEvents()` which groups by `toolName`/`eventType` and returns sorted counts. This is used in session briefings and daily digests as structured input to the LLM prompt, not as a standalone LLM call.
