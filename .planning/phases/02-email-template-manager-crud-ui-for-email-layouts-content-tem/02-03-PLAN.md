<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Astridr API | Layout CRUD and asset upload cross to external API |
| User HTML -> Monaco Editor | User-authored HTML is stored but not rendered in CodePulse DOM (only in iframe preview in Plan 04) |
| User file -> upload | File crosses from local filesystem to Astridr/Supabase Storage |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Tampering | AssetDropzone | mitigate | Client-side validation: max 5MB, image/* types only (png/jpeg/webp). Server-side enforces same limits. Reject before upload call. |
| T-02-06 | Tampering | AssetDropzone | mitigate | FormData upload uses raw Authorization header only — no Content-Type header set. Follows importAgentYaml pattern. |
| T-02-07 | Information Disclosure | LayoutSheet error handling | mitigate | Save/delete errors show generic toast ("Failed to save layout"), never raw API error text. Error object logged to console only. |
| T-02-08 | Tampering | LayoutSheet HTML input | accept | User-authored HTML in header/footer/css is stored via API — the API endpoint handles sanitization. CodePulse does not render this HTML in its own DOM (preview uses sandboxed iframe in Plan 04). |
</threat_model>
