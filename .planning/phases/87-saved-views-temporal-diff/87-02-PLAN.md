<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| URL `?view` param → page hydration | An attacker-controlled `?view` string is read on load and used to look up a Convex view |
| clipboard write | A share URL containing a shareToken is written to the OS clipboard |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-87-05 | Tampering | `?view=<token>` hydration | mitigate | Token is matched by exact equality against `views` (the Convex `getByShareToken`/`list` result); a non-matching/malformed value applies nothing and silently falls back to default state (D-04). The param is never reflected into the DOM, so no XSS via `?view`. |
| T-87-06 | Information Disclosure | shared link config in URL | accept | The shared URL carries only an opaque shareToken (no PII, no filter contents in the URL — config lives Convex-side, D-03). Enumeration of views is accepted per D-02 (global scope). |
| T-87-07 | Elevation of Privilege | ?view vs ?focus double-hydration | mitigate | When a view applies, `appliedFocusRef.current` is set true so the ?focus guard no-ops (Pitfall 5) — prevents an attacker composing `?view&focus` to force an inconsistent lens state. |
</threat_model>
