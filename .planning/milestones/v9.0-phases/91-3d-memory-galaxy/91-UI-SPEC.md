## 3D Node Visual Encoding (D-01a â€” Primary Deliverable of This Spec)

The 2D `paintNode` draws a canvas selection ring and a community halo via `communityColorFn`.
These are canvas-2D primitives and do NOT translate to `react-force-graph-3d`. This section
specifies the 3D-native equivalents using library props only.

### Node State Encoding Table

| State | 2D primitive | 3D-native equivalent | Props |
|-------|-------------|---------------------|-------|
| Default (no selection active) | Theme-colored circle, `shadowBlur=10` | Theme-colored sphere, standard resolution | `nodeColor = colorFn(node)` |
| Hovered (no selection) | White fill `#ffffff`, `shadowBlur=24`, label visible | White sphere, normal size, label tooltip shown | `nodeColor = "#ffffff"` for hovered node |
| Selected node | White fill, `shadowBlur=24`, ring at radius+3, label | White sphere + 1.44Ã— radius bump | `nodeColor = "#ffffff"`, `nodeVal = (node.val ?? 1) * 3` |
| Neighbor of selected | Full opacity, theme color | Full theme color, normal size | `nodeColor = colorFn(node)` |
| Non-neighbor (selection active) | `ctx.globalAlpha = 0.2` (nearly invisible) | Very dark neutral `#27272a` (zinc-800), normal size | `nodeColor = "#27272a"` |

**Selection size bump math:** `nodeRelSize = 4`. Sphere radius = `nodeRelSize * Math.cbrt(nodeVal)`.
Default val=1 â†’ radius=4. Selected val=3 â†’ radius = 4 Ã— âˆ›3 = 4 Ã— 1.44 â‰ˆ 5.76px. This is a
noticeable but not dramatic bump â€” enough to distinguish selected from hovered without jarring
scale shift.

**Dimming implementation note:** `react-force-graph-3d` uses Three.js `MeshLambertMaterial`.
Three.js `Color` does NOT parse `rgba()` strings â€” alpha is ignored. For dimming, use the
solid dark neutral `#27272a` rather than an alpha variant of the theme color. This achieves
the visual effect of non-neighbors receding into the dark backdrop.

**Hover vs. selected distinction:** Both are white. Selected is visually larger (nodeValÃ—3).
When the user hovers over the already-selected node, it renders as white + large (selected wins).
If the user hovers over a non-selected node while another is selected, that hovered node
temporarily shows white + normal size. This is acceptable â€” the size of the selected node
always makes it the focal point.

**Community halo in 3D:** NOT shipped in Phase 91. The `communityColorFn` halo is a canvas-2D
arc+stroke primitive with no three.js equivalent in scope. Sphere color encoding is sufficient.

### Link State Encoding

`react-force-graph-3d` renders links as Three.js `Line` objects. The `linkOpacity` prop
controls a global opacity scalar for all links. Per-link alpha via rgba is NOT supported
(Three.js Color ignores alpha). Link differentiation is achieved via hue (type-based) +
`linkOpacity` global.

| State | Value | Props |
|-------|-------|-------|
| Default codeâ†”code link | `colors.primary` (hex, full) | `linkColor = linkColorFn(link)` |
| Default vaultâ†”vault link | `colors.vaultNode` (hex, full) | same |
| Default cross-source link | `"#94a3b8"` (muted-foreground neutral) | same |
| All links â€” global opacity | `0.2` | `linkOpacity = 0.2` |
| Link width | `0.6` | `linkWidth = 0.6` |

The `linkColorFn` for 3D mirrors the hue intent of the 2D `linkColorFn` (code=primary,
vault=vaultNode, cross=neutral) but uses full-hex colors. `linkOpacity=0.2` provides the
semi-transparent appearance that 2D achieves with alpha variants.

No per-link selection-based brightening in this phase (requires custom material/shader work).
The node selection encoding (white sphere + size bump) provides sufficient visual signal.

Source: `CodeVaultGraph.tsx` L133-147 (2D `linkColorFn`), `ForceGraphCanvas.tsx` L309
(`linkWidth ?? 0.6`).

### 3D Sphere Geometry Props

Starting values for the FPS gate checkpoint. Executor tunes empirically against the live
~4,038-node `graphSnapshots` payload during implementation (CONTEXT.md Claude's Discretion).

| Prop | Starting value | Rationale |
|------|---------------|-----------|
| `nodeResolution` | `6` | Default is 8; reducing to 6 lowers poly count ~44% per sphere â€” primary FPS lever at 4k nodes |
| `nodeRelSize` | `4` | Matches 2D visual scale (`size = Math.max(node.val ?? 3, 2)` ~ 3-4 units) |
| `nodeVal` (default) | `node.val ?? 1` | Inherits the existing `val` field on graph nodes |
| `nodeVal` (selected) | `(node.val ?? 1) * 3` | 1.44Ã— radius bump for selection emphasis |
| `cooldownTicks` | `150` | Gives simulation time to settle at 4k nodes (2D uses 120 cooldownTicks) |
| `warmupTicks` | `0` | Render immediately; simulation settles visually over ~2-3 seconds |
| `linkOpacity` | `0.2` | Global link transparency matching 2D 0.18-alpha intent |
| `linkWidth` | `0.6` | Identical to 2D `linkWidthFn ?? (() => 0.6)` |
| `d3VelocityDecay` | `0.3` | Matches 2D `d3VelocityDecay={0.3}` in ForceGraphCanvas |
| `backgroundColor` | `"#09090b"` | Dark space backdrop; matches 2D `bg-[#09090b]` canvasClass |

- No spinner icon â€” the text-pulse pattern matches the existing snapshot-loading state

### 3D Simulation Stabilization (after chunk loads, before layout settles)

No additional loading UI. The graph renders immediately and the force simulation settles
over ~2-3 seconds with `cooldownTicks=150`. On `onEngineStop`, call `zoomToFit(400, 60)`
matching the 2D `onEngineStop` handler at `CodeVaultGraph.tsx` L489.

### WebGL Disposal on Mode Switch

When switching from 3D â†’ 2D, the `<ForceGraph3D>` component unmounts. The WebGL context
must be disposed to prevent memory leaks (SC#4). Implementation: call
`graphRef.current?._destructor?.()` in a `useEffect` cleanup, or rely on the library's
built-in cleanup if it handles disposal on unmount. Verified via DevTools Memory snapshot
(no orphaned WebGL contexts after 3+ toggles).

| Value type | `"2d" \| "3d"` |
| Default (key absent or read error) | `"2d"` |

## Interaction Parity Contract (D-01)

The following interactions are owned by `GraphContent` and must remain fully functional
in 3D mode without modification to the interaction shell. Only the render surface swaps.

| Interaction | 2D path | 3D requirement |
|-------------|---------|----------------|
| Node click â†’ detail panel | `onNodeClick â†’ setSelectedNodeId(node.id)` | Same callback; `react-force-graph-3d` `onNodeClick` calls identical setter |
| Source filter (Code/Vault/Both) | `filteredData` memo â†’ `<ForceGraphCanvas data={filteredData}>` | Same `filteredData` memo â†’ `<ForceGraph3D graphData={filteredData}>` |
| Detail panel close | `setSelectedNodeId(null)` | Identical |
| Fullscreen toggle | `setFullscreen` â†’ `canvasClass` / `containerClass` change | `canvasClass` applied to 3D wrapper div; same class string |
| ESC to exit fullscreen | `window.addEventListener("keydown")` effect | No change |
| `?focus=` deep-link centering | `useFocusParam` â†’ `centerNodeWhenReady(fgRef, node)` (2D `fgRef.zoomToFit`) | 3D branch: `fgRef3d.current?.cameraPosition({x: node.x, y: node.y, z: node.z ?? 0}, null, 800)` then `zoomToFit(400, 60)` â€” see D-01b in CONTEXT.md; ref type differs |
| KG cross-graph links | Detail panel section, no render dependency | Same JSX, unchanged |
| Neighbor navigation | Detail panel button list | Same JSX, unchanged |
| idb-keyval persistence | n/a (new this phase) | Persist and restore `renderMode` on mount |
| Background click â†’ deselect | `onBackgroundClick â†’ setSelectedNodeId(null)` | `react-force-graph-3d` `onBackgroundClick` â†’ same setter |

**Ref abstraction note (D-01b):** The 2D `ForceGraphHandle` interface (`centerAt`, `zoom`,
`zoomToFit`, `d3Force`, `d3ReheatSimulation`) is defined in `ForceGraphCanvas.tsx` L30-38.
The 3D ref exposes different methods. The planner must wire a mode-conditional ref â€” either
a union type or separate `fgRef2d` / `fgRef3d` refs with a conditional call site for the
focus-param centering path. This is an implementation concern, not a visual contract concern.
