<migration_rules>
<!-- Shared swap contract for Category A. Source: RESEARCH §Category A, PATTERNS §Tailwind arbitrary shadow → CSS variable token. -->

Glow-intensity to token mapping (by blur radius / opacity, choose closest):
  shadow-[0_0_8px_rgba(...0.05..0.1)]   becomes shadow-[var(--glow-xs)]
  shadow-[0_0_15px_rgba(...)]           becomes shadow-[var(--glow-sm)]
  shadow-[0_0_20-25px_rgba(...)]        becomes shadow-[var(--glow-md)]
  shadow-[0_0_40px_rgba(...)]           becomes shadow-[var(--glow-lg)]
  hover:shadow-[...]                     becomes hover:shadow-[var(--glow-sm|md)]
  drop-shadow-[0_0_8px_rgba(...0.8)]    becomes drop-shadow-[var(--glow-xs)] (nearest glow token)

Solid color literals used as brand accent (text/border/bg) with cyan/emerald become the
matching token utility: text-primary / border-primary / bg-primary (or bg-[var(--primary)]).

EXEMPT — DO NOT MIGRATE (locked decision):
  - Per-provider / per-event-type identity color maps (not in this plan's file list)
  - Skills category swatch maps (data-driven identity, not chrome)
  - Any hex that is a data-driven series/identity color (chart series, status semantics keyed
    to data) stays; only glow/shadow/chrome accents migrate.

Readable theme sets --glow-*: none, so a migrated shadow renders no shadow there (valid CSS).
</migration_rules>
</context>
