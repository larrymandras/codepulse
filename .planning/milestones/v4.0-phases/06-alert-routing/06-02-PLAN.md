Create `convex/alertRuleCustom.ts` with these exports:

**`create` mutation** (per D-02):
- Args: `name: v.string(), severity: v.string(), conditions: v.array(v.object({ metric: v.string(), operator: v.string(), threshold: v.float64(), lookbackWindow: v.string() })), conditionLogic: v.string(), conditionGroups: v.optional(...same nested structure...), messageTemplate: v.optional(v.string())`
- Inserts into `alertRuleCustom` with `enabled: true, createdAt: Date.now() / 1000, updatedAt: Date.now() / 1000`
- Returns the new document id

**`update` mutation**:
- Args: `id: v.id("alertRuleCustom"), name: v.optional(v.string()), severity: v.optional(v.string()), conditions: v.optional(...), conditionLogic: v.optional(v.string()), conditionGroups: v.optional(...), messageTemplate: v.optional(v.string()), enabled: v.optional(v.boolean())`
- Patches the record, always sets `updatedAt: Date.now() / 1000`

**`remove` mutation**:
- Args: `id: v.id("alertRuleCustom")`
- Deletes the record

**`list` query**:
- Args: `enabled: v.optional(v.boolean()), severity: v.optional(v.string())`
- Returns all matching custom rules. If `enabled` provided, use `by_enabled` index. Otherwise return all.

**`get` query**:
- Args: `id: v.id("alertRuleCustom")`
- Returns single record

**`setThresholdOverride` mutation** (per D-01):
- Args: `ruleId: v.string(), threshold: v.float64(), lookbackWindow: v.string()`
- Queries `agentConfigs` with `by_key` index for key `"alert-rule-override:{ruleId}"`
- If exists: patch `value` and `updatedAt`
- If not: insert new record with `configKey: "alert-rule-override:{ruleId}", value: { threshold, lookbackWindow }, source: "dashboard", updatedAt: Date.now() / 1000`

**`getThresholdOverride` query**:
- Args: `ruleId: v.string()`
- Queries `agentConfigs` by key `"alert-rule-override:{ruleId}"`
- Returns `value` object or `null`

**`listThresholdOverrides` query**:
- Returns all agentConfigs where `configKey` starts with `"alert-rule-override:"`. Since Convex does not support prefix queries natively, collect all agentConfigs and filter client-side, OR use a dedicated query that fetches by known static rule IDs from the alertRules catalog.

  <behavior>
    - acknowledgeAlert: patches alert with status="acknowledged", acknowledgedBy, acknowledgedAt; keeps acknowledged=true for backward compat
    - resolveAlert: patches alert with status="resolved", resolvedAt=now
    - escalateToTask: inserts task with alertId, patches alert with linkedTaskId; uses alert severity as priority
    - muteTarget: inserts alertMutes record with computed expiresAt based on duration
    - unmuteTarget: deletes alertMutes record for target
    - isTargetMuted: returns boolean — false if no record or expiresAt < now
    - listActiveMutes: returns all non-expired mute records
  </behavior>
  <action>
**Create `convex/alertLifecycle.ts`:**

**`acknowledgeAlert` mutation** (per D-09):
- Args: `alertId: v.id("alerts"), acknowledgedBy: v.optional(v.string())`
- Patches alert: `{ status: "acknowledged", acknowledged: true, acknowledgedBy: args.acknowledgedBy ?? "operator", acknowledgedAt: Date.now() / 1000 }`
- The `acknowledged: true` preserves backward compat with existing queries

**`resolveAlert` mutation** (per D-09):
- Args: `alertId: v.id("alerts")`
- Patches alert: `{ status: "resolved", resolvedAt: Date.now() / 1000 }`

**`escalateToTask` mutation** (per D-11):
- Args: `alertId: v.id("alerts"), title: v.string(), description: v.optional(v.string()), priority: v.optional(v.string())`
- Reads alert to get severity
- Maps severity to priority: critical->urgent, error->high, warning->medium, info->low
- If `args.priority` provided, use that instead of mapping
- Inserts into `tasks`: `{ taskId: crypto.randomUUID(), title: args.title, description: args.description, priority, column: "backlog", columnEnteredAt: Date.now() / 1000, createdAt: Date.now() / 1000, alertId: args.alertId }`
- Patches alert: `{ linkedTaskId: taskDocId }`
- Returns taskDocId

**Create `convex/alertMutes.ts`** (per D-10):

**`muteTarget` mutation**:
- Args: `targetType: v.string(), targetId: v.string(), duration: v.string(), mutedBy: v.optional(v.string())`
- Compute `expiresAt`: parse duration string — "15m"->900, "1h"->3600, "4h"->14400, "24h"->86400, "indefinite"->null. Add seconds to `Date.now() / 1000` for non-indefinite.
- Delete any existing mute for same targetType+targetId first (upsert behavior)
- Insert into `alertMutes`: `{ targetType, targetId, duration, expiresAt, mutedBy, createdAt: Date.now() / 1000 }`

**`unmuteTarget` mutation**:
- Args: `targetType: v.string(), targetId: v.string()`
- Query `alertMutes` by `by_target` index, delete the record if found

**`isTargetMuted` internalQuery**:
- Args: `targetType: v.string(), targetId: v.string()`
- Query `alertMutes` by `by_target` index
- If no record: return false
- If `expiresAt` is null (indefinite): return true
- If `expiresAt < Date.now() / 1000`: return false (expired, lazy cleanup)
- Otherwise: return true

**`isTargetMutedPublic` query** — same logic as `isTargetMuted` but exported as public query for UI consumption.

**`listActiveMutes` query**:
- Returns all alertMutes records. Filter in-memory: exclude records where `expiresAt` is not null and `expiresAt < Date.now() / 1000`.

Since Convex mutations need a running backend, tests should validate argument shapes and exported function existence (import verification). Match existing test patterns in the project.
